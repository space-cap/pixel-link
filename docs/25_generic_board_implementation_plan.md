# 📋 공통 게시판 (Generic Board) 아키텍처 구축 계획서

본 문서는 서비스 내에 공지사항, 자유게시판, 1:1 문의, 제휴 문의 등 다양한 형태의 게시판 요구사항을 단 하나의 공통 소스코드와 데이터 테이블로 효율적으로 소화할 수 있는 **공통 게시판 (Generic Board) 아키텍처**의 구현 계획을 정리한 문서입니다.

---

## 💡 게시판 아키텍처 설계 사상에 대한 의견
사용자님께서 제안해 주신 **"하나의 공통 게시판 기능을 만들어 1:1 문의와 제휴 문의까지 확장하여 재사용하는 방식"**은 소프트웨어 설계 공학(DRY 원칙) 관점에서 **매우 훌륭하고 모던한 최상의 접근 방식**입니다!

이 방식의 장점은 다음과 같습니다.
1. **유지보수 비용의 극적인 절감**: 게시글 CRUD(생성, 조회, 수정, 삭제) 및 파일 첨부, 페이징 로직을 게시판 타입별로 중복 코딩할 필요 없이 한 번만 잘 만들어 두면 되므로 후임 개발자가 코드를 파악하기가 매우 수월해집니다.
2. **유연한 확장성**: 나중에 '이벤트 게시판', '업데이트 노트' 등 새로운 형태의 소통 채널이 필요해지더라도 코드를 추가 개발하지 않고 데이터베이스의 타입 구분값(`board_type`)만 추가하여 즉시 서빙할 수 있습니다.
3. **통합 관리 기능**: 모든 게시글 정보가 하나의 아키텍처를 따르기 때문에, 어드민 콘솔(Admin Console)에서 전체 게시판의 최근 문의 현황이나 공지사항 등을 한눈에 통합 관리하고 대시보드 통계를 내기가 매우 유용해집니다.

---

## 1. 데이터베이스(DB) 스키마 설계

로컬 SQLite와 운영 환경 PostgreSQL 모두 호환되도록 표준 컬럼 구조로 설계하며, 1:1 문의의 '비밀글' 설정이나 제휴 문의의 '진행 상태' 등을 유연하게 처리할 수 있는 필드를 포함합니다.

### 1.1. `board_articles` (게시글 테이블)

| 컬럼명 | 데이터 타입 | 제약 조건 | 설명 |
| :--- | :--- | :--- | :--- |
| `id` | VARCHAR(50) | PRIMARY KEY | 게시글 고유 식별 ID (UUID 등) |
| `board_type` | VARCHAR(50) | NOT NULL | 게시판 구분 (`NOTICE`, `FREE`, `QNA`, `PARTNERSHIP` 등) |
| `title` | VARCHAR(255) | NOT NULL | 게시글 제목 |
| `content` | TEXT | NOT NULL | 게시글 본문 내용 |
| `author_id` | VARCHAR(50) | NULL | 작성자 ID (`users.id` 외래키, 비회원 작성 시 NULL 허용) |
| `author_name` | VARCHAR(100) | NOT NULL | 작성자 노출 이름 |
| `is_secret` | BOOLEAN/INT | DEFAULT FALSE | 비밀글 여부 (1:1 문의용, 본인 및 관리자만 조회 가능) |
| `password` | VARCHAR(255) | NULL | 비회원 작성글 수정을 위한 암호화된 비밀번호 |
| `status` | VARCHAR(50) | DEFAULT 'OPEN' | 처리 상태 (`OPEN`, `RESOLVED` (답변완료), `PENDING` (제휴검토중)) |
| `view_count` | INTEGER | DEFAULT 0 | 조회수 |
| `created_at` | TIMESTAMP | DEFAULT CURRENT_TIMESTAMP | 최초 작성일시 |
| `updated_at` | TIMESTAMP | DEFAULT CURRENT_TIMESTAMP | 최종 수정일시 |

### 1.2. `board_comments` (댓글 및 관리자 답변 테이블)

| 컬럼명 | 데이터 타입 | 제약 조건 | 설명 |
| :--- | :--- | :--- | :--- |
| `id` | VARCHAR(50) | PRIMARY KEY | 댓글 고유 식별 ID |
| `article_id` | VARCHAR(50) | REFERENCES board_articles(id) | 대상 게시글 ID (외래키, cascade delete) |
| `author_id` | VARCHAR(50) | NULL | 작성자 ID (비회원 허용 시 NULL) |
| `author_name` | VARCHAR(100) | NOT NULL | 작성자 노출 이름 |
| `content` | TEXT | NOT NULL | 댓글 및 답변 내용 |
| `is_admin_reply` | BOOLEAN/INT | DEFAULT FALSE | 관리자 공식 답변 여부 (1:1 문의 답변 시 강조 표시용) |
| `created_at` | TIMESTAMP | DEFAULT CURRENT_TIMESTAMP | 작성일시 |

## 2. 세부 구현 로직 및 보안/권한 제어

게시판의 유형(`board_type`)에 따라 권한(Security Filter) 및 조회를 동적으로 통제합니다.

### 2.1. 게시판 타입별 접근 통제 정책 (Role-based Control)

1. **공지사항 (`NOTICE`)**:
   - **조회(Read)**: 로그인 여부 상관없이 누구나 조회 가능 (`permitAll()`).
   - **작성/수정/삭제(Write/Modify)**: 오직 관리자(`ADMIN`) 역할만 허용.
2. **자유게시판 (`FREE`)**:
   - **조회/작성/댓글**: 기본적으로 회원(`USER`, `ADMIN`) 접근 가능. (설정에 따라 비회원 읽기 허용 가능)
3. **1:1 문의 (`QNA`)**:
   - **작성(Write)**: 로그인한 회원 또는 이메일 검증된 비회원만 작성 가능.
   - **조회(Read)**: 비밀글(`is_secret=true`)의 경우, **글을 쓴 본인** 또는 **관리자(`ADMIN`)**만 상세 내용을 열람할 수 있도록 Spring Security 및 MyBatis 쿼리 필터 단에서 엄격히 이중 검증.
4. **제휴 문의 (`PARTNERSHIP`)**:
   - **작성(Write)**: 광고주/협력사 등 비로그인 방문자도 쉽게 신청할 수 있도록 비회원 작성 폼 개방.
   - **조회(Read)**: 비즈니스 정보 유출 방지를 위해 일반 대시보드 목록에는 노출하지 않으며, **오직 어드민 백오피스(/app/admin)에서 관리자만 일괄 조회**하도록 설계.

## 3. 사용자 인터페이스 (UI/UX) 구성안

* **공통 게시판 레이아웃**: 상단 카테고리 탭 스위처를 통해 공지사항, 자유게시판 등을 매끄럽게 전환하며 페이징 처리된 깔끔한 카드 리스트를 노출합니다.
* **1:1 문의 / 제휴 문의 전용 페이지**:
  - 비밀글 자물쇠 체크 옵션을 주어 1:1 문의 전용 작성 폼처럼 연출합니다.
  - 관리자가 어드민 콘솔에서 문의글 하단에 답변 폼을 열어 답글을 작성하면 즉시 상태가 `RESOLVED` (답변 완료) 상태로 전환되고 본문 하단에 관리자 공식 피드백 카드가 렌더링되도록 구성합니다.

## 🔍 의사결정이 필요한 질문 사항 (Open Questions)

> [!IMPORTANT]
> **1. 비회원(로그인하지 않은 방문자)의 글쓰기 허용 여부**
> - **의견 조율**: 제휴 문의 및 1:1 문의의 경우, 아직 로그인하지 않은 예비 광고주나 비회원 사용자도 유입 경로에서 즉각 작성할 수 있도록 비회원 글쓰기(이름, 이메일, 비밀글 패스워드 입력형)를 허용하는 것이 마케팅 효율에 좋습니다. 회원으로만 제한할지, 비회원 폼도 제공할지 의견이 필요합니다.

> [!NOTE]
> **2. 첨부파일(이미지/문서 등) 업로드 기능의 범위**
> - **의견 조율**: 자유게시판에 사진을 첨부하거나 1:1 문의 시 오류 캡처 이미지를 첨부하는 요구사항이 있을 수 있습니다. 초기 버전에는 구현의 속도를 위해 텍스트 입력만 먼저 지원할지, 아니면 초기 버전부터 로컬 디스크 파일 업로드 모듈을 포함하여 설계할지 확인하고 싶습니다.

> [!WARNING]
> **3. 자유게시판 내 비속어 필터링 및 신고 시스템 필요성**
> - **의견 조율**: 자유게시판은 익명 유저나 회원의 불건전한 도배가 있을 수 있습니다. 우선은 관리자가 어드민 콘솔에서 강제로 게시글을 삭제할 수 있는 관리자 삭제 권한만 탑재하고, 차후 서비스 스케일이 커질 때 자동 비속어 금칙어 필터나 사용자 신고 기능을 고도화하는 방향을 추천해 드립니다.
